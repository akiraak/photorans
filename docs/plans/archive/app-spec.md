- 翻訳方向: **英語 → 日本語** (MVP)
- プロジェクト性質: **ストア公開を目指す** (個人開発 → 商用 / セミ商用)
- 競合 / 参考アプリ: _未確定_ (Google レンズ、DeepL カメラ翻訳、Apple 純正翻訳など)

## 3. 機能要件

### コアフロー
1. 写真撮影 (またはギャラリーから選択?)
2. OCR でテキスト抽出
3. 翻訳
4. 結果表示

### 重点対応する書類タイプ (Q4 回答より)
- 郵便書類 (役所からの通知書、銀行・保険等の郵送物全般)
- 医療 (問診票、薬の説明書、検査結果など)
- 契約書類 (賃貸、携帯、金融など)
- 日常 (商品ラベル、食品表示、メニューなど)
- 交通案内 (駅・空港・道路標識、運行情報)

### 追加機能候補 (要否は未確定)
- 履歴 / お気に入り保存
- 複数画像のバッチ処理
- 翻訳結果の音声読み上げ (TTS)
- 翻訳結果のコピー / シェア
- 撮影前の言語選択 vs 自動言語検出

## 4. OCR

- 方式: **自前サーバ経由でクラウド AI を呼び出す** (オンライン必須、オフライン対応なし)
- 候補:
  - **マルチモーダル LLM ワンショット** (Claude / GPT / Gemini に画像を直接渡し OCR + 翻訳を同時実行) ← 第一候補
  - **2 段構成** (Google Cloud Vision / Azure Document Intelligence で OCR → LLM で翻訳)
- 対応言語: 英語のみ (ラテン文字)
- 精度・レイテンシ要件: _未確定_ (目安: 1 枚 3〜5 秒以内、専門用語の誤訳を最小化)

## 5. 翻訳

- バックエンド: **自前サーバ経由のクラウド AI** (OCR と統合、または OCR の後段で LLM 翻訳)
- 対応言語ペア: **英語 → 日本語のみ**。将来の言語拡張は **想定しない**
- ソース言語の自動検出: 不要 (英語固定)
- オフライン対応: **不要** (オンライン必須)

## 7. データ・プライバシー

- 画像を端末外に送るか: **送る** (自前サーバ → クラウド AI)
- 画像・テキストの保管方針: **サーバで即時破棄** (ログにも本文を残さない設計にする)
- 履歴の保存場所: **端末ローカルに全件保存** (SQLite / WatermelonDB / AsyncStorage 等で永続化)
- アカウント / ログインの要否: **完全匿名** (デバイス ID ベース、ログイン不要)
- 利用規約・プライバシーポリシー: 「画像はサーバ経由でクラウド AI に送信され、処理後即時破棄される」ことを明記する

- パフォーマンス目標: 撮影〜翻訳結果表示まで **5 秒以内** を目安 (要確定)
- API 利用コストの上限想定: _未確定_ (1 リクエスト ≒ 数円〜十数円規模 / 課金モデルと連動)
- 多言語化 (アプリ UI 自体): **不要** (UI は日本語のみ)

## 議事メモ

### 2026-04-28
- 当初「日本在住の外国人 / 生活支援」と整理したが、Q5 回答 (英→日) と矛盾。翌日確認することに。
- React Native 採用で iOS / Android 両対応が前提に。OCR は ML Kit / Vision Camera 系プラグイン or クラウド API のどちらを使うか後段で検討。

### 2026-04-29
- ターゲットを **海外在住の日本人** に修正 (Q5 回答 = 英→日 が正)。これに伴い前提が以下のように変わる:
  - **アプリ UI は日本語ベースで OK** (多言語化は不要 or 後回し)
  - **OCR は英語 (ラテン文字) 認識が主**。日本語 OCR は当面不要
    - Apple Vision / Google ML Kit いずれも英語の活字 OCR は実用レベルで無料
    - 手書き (医療の問診票回答など) はやや難しいが、印字書類が主なら問題は少ない
  - **滞在国の通信事情** に依存: 海外渡航初期や低電波下では通信が不安定 → オフライン動作の価値が高い
  - **データプライバシー**: 個人情報書類 (請求書・診断書・契約書) を扱う点は変わらず重要
  - **コスト感**: ユーザー数 = 「海外在住日本人」は限定的 (世帯数十万〜百万規模) なので、クラウド翻訳 API の単価と LTV のバランスは要シビアに見積もる
- 翻訳エンジン候補:
  - **DeepL** — 英→日精度が高いと評価される。API 有料。
  - **Google Translate / Cloud Translation** — 安価で広範。
  - **LLM (Claude / GPT)** — 文脈理解 / 専門用語 (医療・契約) に強い。コストは高め。
  - **Apple 翻訳 / ML Kit 翻訳 (オンデバイス)** — オフライン可。精度は専門用語で劣る。
  - 案: 「オンデバイス翻訳 (オフライン保険) + LLM (専門用語の高精度モード)」のハイブリッドが有力。
- **2026-04-29 確定**: サーバ側 AI 一本化 (オフライン非対応)。これにより以下のシンプル化が可能:
  - クライアントは画像送信 + 結果表示のみ (Vision Camera 系プラグインで撮影→アップロード)
  - サーバで OCR + 翻訳を一括処理 (マルチモーダル LLM ワンショット案 vs OCR API + LLM 翻訳の 2 段構成)
  - **コスト試算 (Claude Sonnet 4.6 想定)**: 画像 1 枚あたり概ね $0.005〜$0.015 (≒ 1〜2 円)。MAU 1000・月 50 枚利用なら原価 $250〜$750 / 月。サブスク or 月間枚数制限が現実的。
  - **アーキテクチャ案**:
    - クライアント (RN) → 自前 API (例: Cloudflare Workers / Vercel / AWS Lambda) → Anthropic / OpenAI / Gemini API
    - 画像は S3 / R2 等に一時保管 or プロセス内で破棄
    - レート制限・認証で API キー流出と濫用を防ぐ
- **2026-04-29 追加確定**:
  - 課金なし / 認証なし / サーバ即時破棄 / 履歴クライアント保存 という最小構成。
  - **濫用リスク**: 認証なしのプロキシは API キー濫用の標的になりうる。緩和策の検討が必要:
    - デバイス ID + 簡易シグネチャ (App Attest / Play Integrity API) によるリクエスト検証
    - IP / デバイス ID ベースのレート制限 (Cloudflare Workers + Durable Objects か Upstash Redis 等)
    - 画像サイズ・MIME タイプの上限
    - サーバ側で月次予算アラート / Anthropic 等の使用量上限を設定
  - **ホスティング第一候補**: Cloudflare Workers + R2 (画像中継用、即時削除) + Durable Objects / KV (レート制限)。完全マネージドで運用負荷ゼロ、無料枠で MVP まではほぼ収まる。
  - **クライアント保存の実装候補**: SQLite (op-sqlite / expo-sqlite) で原文・訳文・サムネイルを保持。検索・並び替え対応のため SQLite が無難。
  - **ストア審査の留意点**:
    - カメラ / 写真ライブラリ権限の用途明記
    - クラウド送信される旨をプライバシーポリシーに明示
    - 「医療相談・契約判断の代替ではない」旨のディスクレーマーが望ましい (誤訳リスクのカバー)

### 2026-04-29 アーキテクチャ確定

```
[iPhone / Android (Expo Dev Client)]
   │  HTTPS or HTTP (LAN)
   │  POST /translate { image (base64 or multipart) }
   ▼
[自宅 Linux PC]
   ├── docker compose
   │     └── api: Node.js (TypeScript) + Hono / Fastify
   │           └─ adapter: ─┬─ Anthropic Claude Sonnet 4.6 (primary)
   │                        └─ Google Gemini 2.5 Flash (fallback / 比較用)
   │  画像はメモリ上で破棄、ログには残さない
   ▼
[クラウド AI API] (Anthropic / Google)
```

- **MVP では LAN 内利用** で十分 (Akira さんが自宅 Wi-Fi 内で実機テストする想定)。
- **将来の外部公開時** は Cloudflare Tunnel / Tailscale Funnel + 共有トークンを後付けすれば対応可能。
- Docker Compose を採用することで、後でホスト先を変えても (VPS / Cloud Run 等) 移植容易。
