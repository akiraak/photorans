# アプリ画面構成の設計

## 目的

photorans の画面構成を再設計し、以下を満たすものにする。

- **コンテンツ単位の確立**: `写真 + オリジナル英文 + 日本語訳` を 1 件のコンテンツ (Item) として扱う
- **コンテンツ一覧をメイン画面化**: 起動直後にコンテンツ一覧が見える
- **グループ化 (ディレクトリ化)**: コンテンツを任意のグループにまとめられる
- **コンテキスト依存のカメラ起動**: 画面に常設のカメラアイコンがあり、現在表示中の画面 (トップ or 特定グループ) に従って保存先が自動で決まる

このプランでは、まず複数の画面構成パターンを洗い出し、長短を比較してユーザーと合意するところまでをスコープとする。実装スキャフォールド (データモデル変更や Navigation 書き換え) はパターン確定後に別プランに切り出す。

## 背景 / 現状

- 現状は `RootView` の `TabView` 2 タブ構成 (`カメラ` / `履歴`)。`RootView.swift:12-27` 参照
- 履歴は `HistoryListView` でフラットな時系列リスト (`HistoryListView.swift:6` の `@Query(sort: \.createdAt, order: .reverse)`)
- 永続モデルは `HistoryEntry` のみで、グループ概念は未導入 (`HistoryEntry.swift`)
- カメラは独立タブで起動し、撮影完了後はタブ切り替えで履歴に戻る (`RootView.swift:14`)

つまり「コンテンツ」「グループ」「コンテキスト依存カメラ」の 3 概念が現状ゼロからの追加になる。

## 用語の前提

このプラン内では以下の語を使う。実装時に英名を確定させる予定。

- **Item** … 1 コンテンツ (写真 + 原文 + 訳文 + メタデータ)。現行の `HistoryEntry` を発展させたもの
- **Group** … Item を束ねる単位。フォルダ / ディレクトリ相当
- **Root (トップ)** … いずれの Group にも属さない Item の集合 (≒ ホーム画面で見える領域)
- **保存コンテキスト** … カメラを起動した画面が Root か特定 Group か、という状態。撮影後に Item をどこへ保存するかを決める

階層は **1 階層フラット** とするか **N 階層ネスト** とするかが論点 (後述パターンで分岐)。

## 要件 (確定)

R1. Item は `写真`, `原文 (英)`, `訳文 (日)` を必須フィールドとする (現状の `HistoryEntry` フィールドを継承)
R2. メイン画面はコンテンツ一覧 (Root の Item / Group が見える)
R3. Group を作成・削除・リネームできる
R4. Item を Group に所属させられる (作成時に自動で / 後から手動で移動)
R5. どの画面 (Root or Group 内) からでも、画面に常設されたカメラボタン 1 タップでカメラ起動
R6. 撮影 → 翻訳完了後の保存先は、カメラを起動した画面のコンテキストで決まる (Root から起動 → Root、Group X から起動 → Group X)
R7. 既存の `HistoryEntry` が入った SwiftData ストアからのマイグレーションが必要 (既存ユーザの撮影結果は Root に紐付ける想定)

## 未確定の項目 (ユーザー判断が必要)

Q1. **Group のネスト**: 1 階層フラットで十分か、N 階層ネスト (Group 内 Group) を許すか
Q2. **Item の複数 Group 所属**: 1 Item は 1 Group のみか、タグのように複数所属可能にするか
Q3. **撮影後フロー**: 撮影 → 翻訳結果プレビュー → 保存ボタンで確定 か、撮影完了即保存して一覧画面に戻る か
Q4. **iPad / 横向き対応**: 当面 iPhone 縦のみで割り切るか (メモリの `feedback_camera_ui_portrait_only` に従い当面は iPhone 縦想定)
Q5. **検索 / 絞り込み**: メイン画面に検索バーや並び替えを今フェーズで入れるか、後回しか

これらはパターン比較のときに「このパターンはこの Q に対する立場が異なる」という形で論点化する。

## 検討する画面構成パターン

代表的な iOS アプリの構成を参考に、4 パターンを挙げる。実装複雑度はあくまで体感の相対比較。

### パターン A: シングルスタック型 (写真アプリ風)

- ルート画面 = Root の Item / Group を 1 つの一覧 (グリッド or リスト) に混在表示
- Group を tap → push 遷移で Group 内の一覧へ
- 画面右下に常設のフローティングカメラボタン (FAB)、現在画面のコンテキストで撮影
- Item を tap → 詳細画面 (写真大 + 原文 + 訳文)
- タブバー無し

長所:
- 一覧がメイン画面という要件 R2 と素直に一致
- 「今いる画面に保存」というメンタルモデルが NavigationStack の階層と完全一致するので R6 が直感的
- iOS の写真アプリ / Files アプリと類似で学習コストが低い

短所:
- 履歴を辿るのと撮影起動が同じ画面に同居するので、撮影頻度が高い場合タップ動線が深くなる可能性 (FAB が常設されているので 1 タップではあるが、Group の奥にいるときに「Root に保存したい」と思うと一旦戻る必要がある)
- Group 切り替え時に Navigation の戻り操作が必要

立場:
- Q1: ネスト (N 階層) と相性が良い (Files アプリ的)
- Q2: 1 Group 1 Item 前提の方がシンプル
- Q3: 撮影後即保存 + 一覧トップに戻る挙動が綺麗

### パターン B: 2 タブ + コンテンツ一覧型

- タブバーは `コンテンツ` / `カメラ` の 2 つ (現状とほぼ同じ構成だが「履歴」を「コンテンツ」に格上げ)
- `コンテンツ` タブ内で Group / Item の階層を NavigationStack で扱う
- カメラタブは現行どおり、保存先は常に Root (or 直前に開いていた Group をタブ切り替え跨ぎで覚える)

長所:
- 現行構成からの差分が最小、移行コストが低い
- カメラへの動線が常にタブバー 1 タップで一定

短所:
- 「表示中の画面で保存先が決まる」R6 とタブ分離が衝突する。タブを切り替えた瞬間にコンテキストが曖昧になる
- 結局 R6 を満たすには「直前の Group を覚える」など暗黙の状態が必要で、ユーザーに伝わりにくい

立場:
- R6 を厳密に守るとパターン B は破綻気味。妥協してでも現行 UI を残したい場合の選択肢

### パターン C: NavigationStack + ボトム常設カメラバー

- タブバー無し、画面下部に「カメラボタン (中央) + 補助ボタン (Group 作成 / 検索)」を持つ常設バーを置く
- メイン領域は NavigationStack の現在ページ (Root or Group 内)。Group へ push してもボトムバーは出続ける
- カメラボタンは常に現在ページのコンテキストで起動

長所:
- R5 / R6 が UI レベルで一目瞭然 (常に画面下にカメラがあり、押した場所がそのまま保存先)
- パターン A の FAB より「主操作」感が強く、撮影中心のアプリらしい見た目
- 純正カメラアプリのシャッターボタンメタファに近い

短所:
- ボトムバーを自前で組む必要があり、SwiftUI の `.toolbar(.bottomBar)` か `safeAreaInset` で実装する手間がある
- セーフエリアと NavigationStack の `.toolbar` の競合を意識しないと、Group 詳細などで重なる

立場:
- Q1: フラット 1 階層と相性が良い (深いネストはボトムバー UI と相性悪い)
- Q3: 撮影 → モーダルで翻訳確認 → 保存ボタン、の方が UI 一貫性がある

### パターン D: サイドバー / Sidebar + カラム型 (iPad 視野)

- `NavigationSplitView` でサイドバーに Group リスト (+ Root)、メインカラムに選択中 Group の Item 一覧
- カメラボタンはメインカラムのツールバー右上に常設
- iPhone では自動で stack 表示にフォールバック

長所:
- iPad / Mac Catalyst を将来見据えるなら自然
- Group の切り替えがサイドバー 1 タップで済む
- R6 はメインカラムが現在 Group を保持しているので明確

短所:
- iPhone 単体だと NavigationSplitView の stack フォールバックは結局パターン B + A の中間になり、メリットが薄い
- 当面 iPhone 縦のみ (Q4) であれば過剰投資

立場:
- iPad/Mac 対応を真剣に視野に入れる時のみ第一候補

## ホーム画面で Group が埋もれる問題

パターン A 素案 (Group と未分類 Item を同列に並べる) は、未分類撮影が増えると Group が下に押しやられる致命的な弱点がある。Root の表示構造には以下のサブパターンが考えられる。これは A / C / D いずれを採用しても **Root の中身の組み方** として独立に選べる軸。

### サブパターン H1: 上下セクション分離

- Root を 1 つのスクロールビュー内に 2 セクション
  - 上: 「グループ」セクション (常に展開、2 列グリッド、各 Group のカバー画像 + 名前 + 件数)
  - 下: 「未分類」セクション (時系列リストまたはグリッド)
- セクション境界に sticky header を付け、未分類が長くなっても Group ヘッダーは上端に視覚的アンカーが残る

長所:
- 「Group はある」事実が常に見える
- 実装が直感的 (`List` の `Section` か `LazyVGrid` の二段組)
- 未分類を見せ続ける ≒ R6 の「Root に保存される撮影」結果を即確認できる

短所:
- 未分類が大量だとスクロールで Group セクションが画面外に出る (sticky header で部分緩和)
- Group 数が増えるとグリッドが縦に伸び、未分類セクションに到達する前にスクロールが必要

### サブパターン H2: インボックス型 (未分類も特別 Group)

- Root には **Group のみ** が並ぶ。未分類 Item は「インボックス」という特別な Group として常に先頭にピン留め (削除不可、リネーム不可)
- Root から撮影 → 自動的に「インボックス」へ保存。後から Group へ移動できる
- 未分類 Item を直接 Root に出さないので埋もれが構造的に発生しない

長所:
- データモデルが綺麗 (Item は必ず何らかの Group に属する。Root の Item 直下リレーションが不要)
- iOS 純正アプリ (メモ / リマインダー / メール) で実証済みのメタファ
- Group 数が多くてもスクロールするだけで全 Group を一覧できる
- 「インボックスを後で整理する」ワークフローと自然に噛み合う

短所:
- ユーザーから見ると「Root に直接保存」という素朴なメンタルモデルから一段抽象化される (要件 R6 の「Root に保存」の解釈変更)
- 撮ってすぐ翻訳結果を見たい場合、Root → インボックス → Item 詳細、と 1 画面深い

### サブパターン H3: ヘッダーピン (横スクロール) + 本文未分類

- Root 上部に Group の横スクロールカードレーン (高さ 100pt 程度、固定表示)
- 本文は未分類 Item の縦リスト/グリッド
- 「すべて見る」ボタンで Group 全画面リストへ

長所:
- Group が常に画面上部に見える (スクロールで未分類を追っても消えない)
- 撮影直後の未分類 Item が即座に本文最上部に出るため確認動線が短い

短所:
- 横スクロールカードは Group が増えると右にどんどん隠れる (10 個超で破綻気味)
- 「ヘッダー = Group / 本文 = Item」の二重構造を初見ユーザーに伝える必要がある

### サブパターン H4: セグメント切替

- Root 上部に `[グループ | 未分類 | すべて]` のセグメントコントロール、デフォルトは `グループ`
- 各セグメントの内容を切り替えて表示
- 撮影後は自動で「未分類」セグメントへフォーカスして結果を見せる

長所:
- それぞれの View が単純な縦リストで実装しやすい
- Group 一覧は未分類量に左右されず常に一定の見え方

短所:
- 「現在画面のコンテキストでカメラ起動 (R6)」が曖昧化する (「グループ」セグメント表示中に撮ったら未分類に行くのか?)
- セグメント切替がもう一つの主操作になり、画面下のカメラと注意が分散

### サブパターン H5: Group 優先 + 未分類折りたたみ

- 上に Group リスト (常に全件)、下に「未分類 (N)」セクションをデフォルトで折りたたみ表示
- ヘッダータップで展開/折り畳み
- 撮影直後だけ自動展開して新しい Item をハイライト

長所:
- H1 の弱点 (未分類が大量で Group が押しやられる) を構造的に解消
- 折り畳み状態でも件数が見える

短所:
- 「未分類が見えない」ことに最初戸惑う可能性
- 折り畳み状態の管理を覚えるか毎回リセットするかの設計判断が必要

### サブパターン比較

| Group 埋もれ防止 | ○ (sticky) | ◎ (構造的) | ○ (上部固定) | ◎ | ◎ |

## 推奨 (たたき台、改訂版)

画面構成は引き続き **パターン A (シングルスタック + FAB)** を第一候補とし、Root の中身は **サブパターン H2 (インボックス型)** を組み合わせるのが最もバランスが良い。

理由:
- H2 は Group 埋もれを構造的に防ぐ唯一のパターン (他は緩和策)
- データモデル面で「Item は必ず 1 つの Group に属する」と一様化でき、SwiftData リレーションが綺麗 (`Group (1) ↔ (N) Item` だけで済み、Root 直下 Item の特例不要)
- 「インボックス」は削除不可・固定先頭のシステム Group として実装、ユーザーから見ると「未分類置き場」
- R6 の解釈は **「Root から撮影 = インボックスへ保存」「Group 内から撮影 = その Group へ保存」** と再定義する。ユーザー体験はほぼ変わらず、内部モデルだけ綺麗になる
- メモ / リマインダー / メールと同じメタファなので学習コストも低い

第二候補は **A + H1 (セクション分離)**。H2 の「1 階層深い」を嫌う場合の素直な代替。撮影直後の Item が Root に直接見えるため、撮って即確認の体感は H2 より良い。

第三候補は **C + H2** (ボトムバー + インボックス)。撮影中心 UI を強調したい場合に再評価。

## 決定 (2026-05-02)

ユーザー判断により **Root 構成は H4 (セグメント切替型)** を採用。詳細仕様は別プラン `docs/plans/home-screen-h4-spec.md` で詰める。

画面構成側 (パターン A / C / D) はこのプラン段階では未確定のまま H4 詳細設計フェーズに入り、H4 の保存先解決 (R6) の議論と合わせて確定させる。
