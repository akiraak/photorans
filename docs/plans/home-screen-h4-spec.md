# ホーム画面 H4 (セグメント切替型) 詳細設計

## 採用方針

`docs/plans/screen-architecture.md` の比較で選定された **サブパターン H4 (セグメント切替型)** をベースに、Root 画面の振る舞いとデータモデルを確定させる。

H4 の素案:
- Root 上部に `[グループ | 未分類 | すべて]` のセグメントコントロール
- セグメントを切り替えると本文の表示内容が切り替わる
- 撮影直後は新しい Item を見せるためのフォーカス制御を入れる (詳細はこのプランで詰める)

## このプランのスコープ

H4 の振る舞いを実装可能なレベルまで詰める。コード変更はまだ行わない。仕様確定後に別フェーズで実装タスクを TODO に展開する。

含む:
- データモデル (`Group`, `Item`)
- セグメント仕様 (構成、デフォルト、状態保存)
- 各セグメントの表示仕様
- カメラ起動 / 保存先解決 (R6)
- Group / Item の CRUD 動線
- 撮影完了後の遷移
- 既存 `HistoryEntry` のマイグレーション

含まない:
- 画面構成パターン (A / C / D) の最終確定 — H4 と並行して詰める
- 検索 / 並び替え / 多言語化 — 後続フェーズ

## 用語

- **Item**: 1 コンテンツ (写真 + 原文 + 訳文 + メタデータ)。現行 `HistoryEntry` の発展形
- **Group**: Item を束ねるユーザー定義の単位
- **未分類**: いずれの Group にも属さない Item の集合 (実装は要決定: nil リレーション or 特別 Group)
- **セグメント**: Root 画面上部のセグメントコントロール (`グループ` / `未分類` / `すべて`)

## 論点と決定 (このプランで埋めていく)

### S1. セグメント構成

**論点**: 上部セグメントは何項目で、ラベルは何にするか。

候補:
- (a) 3 項目 `[グループ | 未分類 | すべて]` (素案)
- (b) 2 項目 `[グループ | 未分類]` (「すべて」は省略、検索で代替)
- (c) 4 項目 `[グループ | 未分類 | すべて | 最近]` (「最近」= 直近 N 日の横断ビュー)

決定: ☑ **(b) 2 項目 `[グループ | 未分類]`** (2026-05-02)

判断材料:
- 「すべて」は時系列の横断ビューだが、Group / 未分類のどちらでも個別に時系列で見られるなら冗長になりがち
- 「最近」は撮影直後の確認動線として有用だが、撮影完了後の自動遷移 (S6) でカバー可能なら不要

### S2. デフォルトセグメント

**論点**: アプリ起動 / Root 復帰時にどのセグメントを表示するか。

候補:
- (a) 常に `グループ` (素案)
- (b) 常に `未分類` (撮影直後の確認重視)
- (c) 前回終了時のセグメントを記憶
- (d) 状況依存: Group が 0 件なら `未分類`、それ以外は前回 / `グループ`

決定: ☑ **(b) 常に `未分類`** (2026-05-02)

### S3. 各セグメントの表示仕様

**論点**: それぞれのセグメントで本文に何をどう並べるか。

#### S3-1. `グループ` セグメント

候補:
- (a) 縦リスト (Group 名 + 件数 + サムネ 1 枚)
- (b) 2 列グリッド (カバー画像 + Group 名 + 件数)

決定: ☑ **(a) 縦リスト** (2026-05-02)

サブ論点:
- カバー画像: ☑ **最新 Item の写真を自動採用** (2026-05-02)
- 並び順: ☑ **更新日時順** (Group 内 Item の最新 createdAt を比較。Group 自体の編集日時ではなく中身の鮮度) (2026-05-02)

#### S3-2. `未分類` セグメント

候補:
- (a) 時系列リスト (1 行 = サムネ + 訳文先頭 + 撮影日時)、現行 `HistoryListView` 相当
- (b) 時系列グリッド (写真アプリ風)
- (c) リスト / グリッド 切替可能

決定: ☑ **(a) 時系列リスト** (2026-05-02)

サブ論点: ☑ **特別なアクションは追加しない** (2026-05-02)
- スワイプアクションも一括選択モードも入れず、行タップで詳細画面に遷移するだけ
- 削除と Group 移動は詳細画面側の動線に集約 (S8 で具体化)

#### S3-3. `すべて` セグメント

S1 で `すべて` を不採用としたため対象外。スキップ。

### S4. カメラ起動 / 保存先解決 (R6 の H4 適用)

**論点**: H4 はセグメント切替で表示内容が変わる。Root から撮影したとき保存先をどう決めるか。

候補:
- (a) Root から起動した撮影は **常に未分類** (セグメントに関係なく)。Group へは Group 詳細画面から起動した撮影が入る
- (b) 表示中セグメントに従う: `グループ` 表示中の撮影は…どこ? 曖昧 → このパターンは破綻気味
- (c) `グループ` セグメント表示中に撮影しようとしたら、保存先 Group を選ぶシートを出す (= 撮影前にユーザーに聞く)
- (d) `グループ` セグメント表示中はカメラボタンを無効化 (撮影には `未分類` か Group 詳細へ移動が必要)

決定: ☑ **(a) Root からの撮影は常に未分類** (2026-05-02)
- R6 を「Root 撮影 = 未分類、Group 詳細撮影 = その Group」と一律解釈
- セグメントに関係なく一貫挙動
- ☑ **G**: Group 詳細画面のカメラ FAB 撮影は当該 Group に保存 (R6 の Group 側適用、2026-05-02 確定)

### S5. カメラボタンの配置

**論点**: H4 セグメントは Root 画面上部に出る。カメラボタン (画面構成パターン A: FAB / C: ボトムバー) の配置と H4 の関係。

候補:
- (a) パターン A 採用: 右下 FAB (セグメントとは独立)
- (b) パターン C 採用: 画面下部に常設バー (セグメントとは独立)
- (c) セグメントバーの右端にカメラアイコンを統合 (1 行に収める)

決定: ☑ **(a) パターン A 採用 / 右下 FAB** (2026-05-02)
- これにより画面構成パターンも A 確定 (TabView は廃止し NavigationStack 単一)

### S6. 撮影完了後の遷移 (楽観的 UI)

**論点**: 撮影 → 翻訳完了後、ユーザーをどこに戻すか。

候補:
- (a) Root に戻り、`未分類` セグメントを自動表示し、新しい Item をハイライト
- (b) 撮影結果の詳細画面を直接表示 (戻ると Root の `未分類`)
- (c) 撮影モーダル内で結果プレビュー → 「保存」ボタンで Root に戻る (現状の `CameraView(onTranslated:)` 相当)
- (d) Group 内から撮影した場合は Group 詳細にとどまり、未分類撮影の場合は (a)

決定: ☑ **楽観的 UI (新案)** (2026-05-02)

挙動:
1. シャッターを切った瞬間にカメラ画面を dismiss し、撮影元 (Root の `未分類` セグメント or Group 詳細) に戻る
2. 撮影元のリスト先頭に **処理中 Item** を即時表示。サムネは撮影写真、テキスト領域はプレースホルダ + アニメーション (シマー or プログレスインジケータ) で「処理中」と分かるように
3. OCR + 翻訳が完了したら処理中 Item を本物の Item にインプレース更新 (アニメーションで遷移)
4. 失敗時の挙動はサブ論点 (下記)

サブ論点 (決定済み):

a. **処理中 Item の永続化**: ☑ **(i) SwiftData に最初から保存** (2026-05-02、B 確定)
   - `originalText` / `translatedText` を `String?` に変更し、翻訳完了時に埋める
   - `status: ItemStatus` enum も併設し、`.processing` / `.completed` / `.failed` を明示
   - アプリ kill → 再起動後も処理中 Item が残り、起動時に未完了 Item を再開する (実装方法は実装プランで詳細化)

b. **アニメーション種別**: ☑ **シマーオーバーレイ** (2026-05-02、D 「まかせる」のため推奨採用)
   - 写真サムネ + テキスト領域の上に薄い斜めグラデーションを流す
   - 失敗時はシマー停止し失敗 UI に切り替え

c. **失敗時の見え方**: ☑ **「失敗」バッジ + リトライボタン** (2026-05-02、C 確定)
   - 処理中 Item に赤バッジ + 「再試行」アクション
   - リトライは同じ画像で OCR + 翻訳を再実行
   - ユーザーが手動削除するまで失敗 Item は残る

### S7. Group の CRUD 動線

**論点**: Group の作成・リネーム・削除をどこから操作するか。

決定 (2026-05-02):

- **作成**: ☑ **カメラ FAB の上に Group 作成 FAB を積む (二段スタック)**
  - 右下のカメラ FAB と縦に並ぶ
  - タップで Group 名入力シート表示
- **リネーム**: ☑ **Group 詳細画面のツールバー** (rename 項目)
- **削除時の挙動**: ☑ **「削除 / キャンセル」の 2 択 (S13-3 と統一、2026-05-02 改訂)**
  - Group 詳細画面のツールバーから「削除」 → 確認ダイアログ「このグループと中身のコンテンツ N 件を削除します。元に戻せません。」 → 削除 / キャンセル
  - 子 Group がある場合は「このグループと子グループ N 件、合計 M 件のコンテンツを削除します。元に戻せません。」と件数を併記
  - 「未分類に移動」「親階層に移動」は提供しない (中身を残したいなら削除前に手動で Item を移動してもらう)
  - SwiftData は配下を再帰的にカスケード削除する

サブ論点 (決定済み):
- **Group 作成 FAB の表示範囲**: ☑ **全セグメントで常時表示** (2026-05-02、A 確定)

### S8. Item の CRUD / 移動

**論点**: Item の削除・Group 移動の動線。

決定 (2026-05-02、「実装が簡単な方」方針):

- **削除**: ☑ **Item 詳細画面のツールバーから「削除」** (確認ダイアログ付き)
  - 一覧側のスワイプアクションは S3-2 で「特別アクション無し」と決めたため入れない
- **Group 移動**: ☑ **(a) Item 詳細画面のツールバーから Group 選択シート** を表示
  - シートには既存 Group のリスト + 「未分類に移動」が並ぶ
  - 一括選択モードは入れない

### S9. データモデル

**論点**: SwiftData モデルの構成。

決定: ☑ **(a) `Item.group: Group?`** (オプショナル、未分類 = nil) (2026-05-02)

確定スキーマ:

```swift
@Model
final class ItemGroup {  // 命名確定 (E = ItemGroup, 2026-05-02)
    @Attribute(.unique) var id: UUID
    var name: String
    var createdAt: Date
    var parent: ItemGroup?  // S13-1: Q1 階層制限なし採用 (2026-05-02)。nil = ルート階層
    @Relationship(deleteRule: .nullify, inverse: \ItemGroup.parent) var children: [ItemGroup]
    @Relationship(deleteRule: .nullify, inverse: \Item.group) var items: [Item]

    init(id: UUID = UUID(), name: String, createdAt: Date = Date(), parent: ItemGroup? = nil) {
        self.id = id
        self.name = name
        self.createdAt = createdAt
        self.parent = parent
    }
}

@Model
final class Item {
    @Attribute(.unique) var id: UUID
    var createdAt: Date
    var imagePath: String
    var originalText: String?    // 処理中/失敗中は nil、翻訳完了時に埋まる
    var translatedText: String?  // 同上
    var model: String?            // 翻訳完了時に埋まる
    var group: ItemGroup?         // nil = 未分類
    var status: ItemStatus        // .processing / .completed / .failed
    var failureReason: String?    // .failed のとき UI 表示 + リトライ判断に使用

    init(...) { ... }
}

enum ItemStatus: String, Codable {
    case processing
    case completed
    case failed
}
```

備考:
- S3-1 の「Group 並び順 = 更新日時順」は Group モデルに専用フィールドを置かず、`items.map { $0.createdAt }.max()` で算出
- S7 の削除時 3 択は SwiftData の deleteRule では表現できないため、UI 側で「nullify (未分類へ) / cascade (Item ごと削除)」を分岐処理
- `originalText` / `translatedText` を optional 化したことで、表示側は `status` で分岐 (`.processing` → シマー / `.completed` → 本文 / `.failed` → 失敗バッジ + リトライ) を行う
- 起動時に `status == .processing` の Item を検出し、未完了として翻訳パイプラインを再投入する (kill 復帰対応)。詳細は実装プラン

### S10. 既存 `HistoryEntry` のマイグレーション

**論点**: 現行 SwiftData ストアに入っている `HistoryEntry` をどう移行するか。

決定: ☑ **(b) 旧データ破棄で作り直し** (2026-05-02、F 確定)

実装方針:
- `HistoryEntry` モデルファイルを削除し、`Item` + `ItemGroup` の新スキーマで `ModelContainer` を構築
- 既存ストアと新スキーマが互換しないため SwiftData は起動時にコンテナ作成に失敗する。これを検知してストアファイルごと削除 → 再作成、というフォールバックを 1 度だけ実行する
- もしくは新ストア URL に切り替え (`HistoryEntry.sqlite` → `Photorans.sqlite` 等) し、旧ファイルは放置 (アプリアンインストール時に消える)
- 同時に Documents 配下の旧画像ファイル (`photos/<uuid>.jpg`) も全削除する (孤児画像が残らないように)
- TestFlight 配信前に Akira さんへ「次のビルドで履歴が全消去されます」と事前通知する

### S13. 階層化 (Q1) に伴う追加論点

Q1 で階層制限なしを採用したため、以下の項目に追加の設計判断が必要。

#### S13-1. データモデル拡張

`ItemGroup` に親リレーションを追加する。

```swift
@Model final class ItemGroup {
    @Attribute(.unique) var id: UUID
    var name: String
    var createdAt: Date
    var parent: ItemGroup?  // nil = ルート階層
    @Relationship(deleteRule: .nullify, inverse: \ItemGroup.parent) var children: [ItemGroup]
    @Relationship(deleteRule: .nullify, inverse: \Item.group) var items: [Item]
}
```

`Item.group` の意味も拡張: nil なら「ルート階層の未分類」、特定の `ItemGroup` を指していればその Group 直下の Item。

#### S13-2. Group 詳細画面の構造

階層を許すと Group 詳細画面は「子 Group」と「Item」を両方扱う必要がある。

候補:
- (a) Group 詳細にも Root と同じセグメント `[グループ | 未分類]` を適用
  - `グループ` セグメント = 当該 Group の `children` 一覧
  - `未分類` セグメント = 当該 Group の `items` 一覧
  - Root と Group 詳細で UI 構造が完全に一致 → 学習コストが低い
- (b) セグメントラベルを `[グループ | アイテム]` に変更
- (c) Group 詳細はセグメント無しで、子 Group + Item の混在リスト

決定: ☑ **(a) `[グループ | 未分類]` を Root / Group 詳細とも共通で使用** (2026-05-02)
- 「未分類」の意味は文脈で読み替え:
  - Root: `Item.group == nil` (どの Group にも属さない Item)
  - Group X 詳細: `Item.group == X` (X 直下にあり、子 Group には属さない Item)
- どちらも「現在の階層直下にあって、(子) Group にまとめられていないもの」という意味で一貫

#### S13-3. Group 削除時の挙動 (子 Group の有無に関わらず統一)

決定: ☑ **「削除 / キャンセル」の 2 択** (2026-05-02、S7 と統一済み)
- 削除を選んだら、その Group・配下のすべての子 Group・配下のすべての Item を再帰的に削除
- 「親階層に移動」「未分類に移動」のオプションは設けない
- ダイアログのメッセージは件数を明示
- 子 Group なしの場合: 「このグループと中身のコンテンツ N 件を削除します。元に戻せません。」
- 子 Group ありの場合: 「このグループと子グループ N 件、合計 M 件のコンテンツを削除します。元に戻せません。」

#### S13-4. カメラ FAB の保存先

S4 / G の延長で、現在表示中の階層に保存する。

決定: ☑ **現在表示中の階層 (Root → 未分類、Group X 詳細 → X 直下) に保存** (2026-05-02)

#### S13-5. Group 作成 FAB の保存先

階層化により、新しい Group をどこに作るかも選択肢が出る。

決定: ☑ **(a) 現在表示中の階層直下に作る** (2026-05-02)
- Root 表示中 → ルート Group として作成
- Group X 詳細表示中 → X の子 Group として作成
- カメラ FAB の保存先 (S13-4) と同じ「現在階層」ルール

#### S13-6. Group の移動 (任意の階層に移す)

決定: ☑ **(b) 移動なし** (2026-05-02)
- Group 詳細ツールバーに「移動」アクションは設けない
- 間違えたら削除して作り直す運用
- 必要性が出てきたら後続フェーズで追加

### S14. 検索 (Q5)

#### S14-1. 検索の起動位置

決定: ☑ **(b) SwiftUI 標準の `.searchable`** (2026-05-02)
- `グループ` / `未分類` セグメントの上にプルダウン式の検索バーを置く
- プルダウンで隠せるので常設の邪魔にならない

#### S14-2. 検索範囲

決定: ☑ **未分類 (Item テキスト) は全件フラット検索 / グループ (Group 名) は現在表示中の階層配下のみ** (2026-05-02)

詳細:
